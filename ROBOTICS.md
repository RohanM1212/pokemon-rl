The fundamental challenge of reinforcement learning is defining what good behavior looks like mathematically. When you build an autonomous agent, you cannot give it a step by step manual or code a rigid decision tree for every possible scenario. The state space is simply too large for that to work. Instead, you have to assign numerical values to outcomes, giving points for good results and taking them away for bad ones, then leaving the algorithm to discover the path on its own. The code does not know what the game or the task actually is. It only knows how to maximize a scalar value over time.
This is identical to the core challenge in modern robotics. When researchers at Boston Dynamics or CMU train a quadruped robot to walk across rough terrain, they do not manually program the trajectory of every joint and actuator. If they did, the robot would fall over the second it hit an unexpected rock. They define a reward function instead. They give the system points for keeping its torso upright, moving forward at a specific velocity, and minimizing unnecessary power draw. The robot figures out the exact mechanics of walking through trial and error. The reward function itself is the ultimate design decision, and everything else is just the computer engine processing the numbers.
This project was direct practice for that exact engineering problem. I built three distinct versions of a reward function, which meant creating three completely different definitions of what winning looks like. Each iteration changed the entire policy of the network. Version 1 gave a naive baseline that taught the agent to fight by balancing damage in versus damage out. Version 2 added a time element and taught it to fight fast. Version 3 introduced environmental context and tried to teach it to fight smart by tracking type matchups. The gap between what I intended to teach and what the agent actually ended up doing was the most interesting part of the entire development process.
What I learned is that the hardest part of reward function design is not the underlying mathematics. It is being precise enough about your constraints so that the algorithm cannot find a loophole you did not intend. This optimization failure is called reward hacking. I saw this clearly when testing version 2. By granting a speed bonus for unused turns, I wanted to encourage clean, efficient victories. Instead, the agent hacked the intent of the code. It realized that maximizing the remaining turn counter yielded more points than playing defensively, so it began completely ignoring its own health bar. It would intentionally take devastating hits just to secure a marginally faster knockout. The math allowed a reckless strategy because I failed to penalize risk.
The point of this project was never to build a perfect Pokemon AI. It exists to give me hands-on experience with the hardest design problem in autonomous systems before I ever start working on physical robots.
